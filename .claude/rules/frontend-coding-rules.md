# 実装ルール

## 0. 基本方針

- 本ルールは常に最優先で適用すること。
- ルール間に矛盾がある場合は、より安全で、より保守性が高く、より共通化を優先する方を採用すること。
- 他ルールや既存仕様と矛盾し、ルール準拠の実装が不可能な場合は、実装を続行せず中断すること。
- 中断時は、以下を必ず出力すること。
  - 矛盾しているルールまたは仕様
  - 続行できない理由
  - 回避案または修正候補
- 既存実装が本ルールに違反している場合は、違反内容を明示した上で、ルール準拠の実装へ修正すること。

## 1. 定数管理

### 1-1. 基本原則

- 定数のべた書きを禁止する。
- ハードコーディングを禁止する。
- マジックナンバーを禁止する。
- 実装中に意味が明示されていない数値、文字列、真偽値、パス、キー名、閾値、上限値、表示文言、判定値を直接記述してはならない。
- 一度しか使用しない値であっても、業務要件・仕様・セキュリティ・表示制御・外部連携・判定条件に関わる場合は、必ず定数化すること。
- 例外に該当するか不明な場合は、定数化を優先すること。

### 1-2. 定数の分類と配置ルール

- 定数は用途に応じて配置すること。
- 以下の値は、必ず定数ファイル（`*.config.ts`）に定義すること。
  - エラーメッセージ / 画面表示メッセージ / エラーコード / メッセージコード
  - ラベル名 / ページ名 / API パス / ストレージキー
  - 上限件数 / 桁数制限 / タイムアウト値 / リトライ回数
  - ステータス判定値 / フラグ判定用文字列 / ソート条件 / フォーマット種別
  - URL / パス / 環境依存値 / 外部連携識別子
  - 業務要件に依存する数値・文字列・真偽値
  - 仕様変更の可能性がある値 / 意味を名称で明示すべき値
  - 画面表示や制御条件に関係する値

### 1-3. 配置先

- 単一機能内でのみ利用する定数は `features/<feature>` のルートに配置すること。
- 複数機能で利用する定数は `src/config` に配置すること。

### 1-4. 定数ファイルの定義方法

- 定数ファイルを使用する場合は、必ず `enum` で定義すること。
- 同一カテゴリの値は、意味単位で `enum` を分けること。
- `enum` 名およびメンバー名は、値の意味が明確にわかる名称にすること。

### 1-5. ファイル上部 `const` の制限

- ファイル上部に定義してよいのは、そのファイルだけで完結する技術的補助定数に限る。
- 以下の値を、ファイル上部の `const` として定義することを禁止する。
  - エラーメッセージ / 画面表示メッセージ / エラーコード / メッセージコード
  - API レスポンス判定値 / ステータス判定値
  - 業務要件に依存する文言 / 仕様変更の可能性がある値
  - 表示や制御に関わる値 / URL、パス、キー名などの外部連携識別子
- hook、component、utility、service、repository、adapter、model の各ファイル内で、エラーメッセージやエラーコードを直接 `const` 定義してはならない。

### 1-6. 既存定義の再利用

- 実装前に、既存の `*.config.ts` に同義または近義の定義が存在しないか確認すること。
- 既存の共通定義が存在する場合は、必ず再利用すること。
- 同義の定数を新規作成してはならない。
- 新しいエラーメッセージ・エラーコード・表示文言・判定値が必要な場合は、まず共通定数ファイルを作成または更新し、その後に実装側で参照すること。
- 実装ファイル内に値を先に仮置きしてから後で共通化する、という進め方を禁止する。

### 1-7. 条件分岐と比較

- 文字列リテラルによる条件分岐を禁止する。
- 判定に使用する文字列・数値・真偽値は、必ず定数化すること。
- ループ回数、ページサイズ、閾値、待機時間、インデックス境界、フォーマット種別、ソート条件などは直接記述せず、必ず名前付き定数として定義すること。

### 1-8. `0`、`1`、`true`、`false` の扱い

- `0`、`1`、`true`、`false` であっても、文脈上の意味を説明する必要がある場合は定数化すること。
- ただし、以下のように言語仕様上または慣例上自明であり、意味が明確なもののみ例外とする。
  - 配列添字アクセスの `0`
  - インクリメント・デクリメント用途の `1`
  - `length === 0` など空判定で用いる `0`
- 上記例外に該当しない場合は、定数化を優先すること。

### 1-9. 補助ルール

- CSS クラス名、HTML 属性値、入力制約値なども、共通利用または意味を持つ場合は定数化すること。
- 一度しか使用しない値でも、業務要件・仕様・セキュリティ・表示制御に関わる場合は定数ファイルへ切り出すこと。

## 2. HTML 属性の動的生成制限

- ユーザー入力値を基に `aria-label` および `className` を動的生成してはならない。
- テストで一意な識別子が必要な場合は `data-testid` を使用すること。
- `data-testid` は本番ビルドに含めないこと。
- ビルド時に除去される属性に限り、動的生成を許可する。
- 外部入力値を属性値へ埋め込む場合は、用途・出所・安全性を明示し、必要最小限に限定すること。

## 3. 機密情報の保存禁止

- `localStorage` に `token`、`password`、`secret`、`refreshToken`、認証情報、セッション情報、個人情報、機密情報を保存してはならない。
- `sessionStorage`、Cookie、IndexedDB、URL クエリ、HTML 属性、hidden input 等についても、機密情報の保存または露出を避けること。
- 他ルールや既存仕様と矛盾する場合は処理を中断し、失敗ログを出力すること。
- 一時保存が必要な場合でも、機密性の高い情報は保存対象にしないこと。
- 永続化の必要がある値は、その値が機密情報に該当しないことを確認した上で扱うこと。

## 4. 警告対応

- 警告は抑制ではなく解消を優先すること。
- 3 回対応しても解消できない場合に限り、一時的な抑制を許可する。
- 抑制する場合は、該当行の直前に `TODO` コメントを記載すること。
- `TODO` コメントには以下を必ず含めること。
  - 発生している警告内容
  - 暫定対応として実施した抑制内容
  - 恒久対応の検討方針
- 理由のない `eslint-disable`、`ts-ignore`、`@ts-expect-error`、warning suppression を禁止する。
- 警告抑制は最小範囲に限定すること。
- ファイル全体・ブロック全体の包括的な抑制を禁止する。

## 5. XSS 対策

- ユーザー入力値は HTML として解釈せず、必ずテキストとして扱うこと。
- `dangerouslySetInnerHTML` の使用を原則禁止とする。
- やむを得ず HTML を扱う必要がある場合は、外部入力を直接使用せず、厳格にサニタイズされた安全な値のみを扱うこと。
- `<`、`>`、`&` などの特殊文字は、必要に応じて HTML エンティティへエスケープすること。
- 外部入力由来のメッセージを画面表示する場合は、HTML として解釈されない実装にすること。
- 外部入力に依存する URL を出力する場合は、ホワイトリスト方式で検証すること。
- `javascript:`、`data:`、`vbscript:` など、許可していないスキームは使用禁止とする。
- リンクとして許可するのは、`http://` または `https://` で始まる URL のみとする。
- 外部入力を基にリンク先や画像 URL を動的生成する場合は、必ずホワイトリスト方式で検証すること。
- `target="_blank"` を使用する場合は、必ず `rel="noopener noreferrer"` を併用すること。
- 外部入力値を `href`、`src`、`title`、`aria-label` 等に使用する場合は、安全性を確認した上で使用すること。
- API 由来や外部由来の `message`、`detail`、`title` 等をそのまま HTML として埋め込んではならない。

## 6. eval 使用制限

- `eval` の使用は禁止を原則とする。
- `Function` コンストラクタ、文字列からの動的コード実行、同等の危険な実装も禁止を原則とする。
- やむを得ず使用する場合は、以下のいずれかを必須とする。
  - `eval` の引数を構成する値に、外部入力を直接使用しない
  - `eval` の引数を構成する全ての値を検証し、許可済みの処理のみ実行可能にする
- 上記を満たせない場合は実装を中断すること。

## 7. 実装手順

- 実装前に、追加・変更対象の値が以下に該当するか確認すること。
  - エラーメッセージ / エラーコード / 表示文言 / 判定値
  - API パス / ストレージキー / 業務要件依存値
- 該当する場合は、実装ファイル内で直接定義せず、既存の `*.config.ts` を探索すること。
- 既存定義がある場合は必ず再利用すること。
- 既存定義がない場合のみ、適切な `*.config.ts` を新規作成または更新して `enum` を追加すること。
- その後に、実装ファイル側で import して参照すること。
- 実装の途中でルール違反を発見した場合は、違反内容を明示して修正すること。
- ルール違反を回避できない場合は、実装を中断して理由を明示すること。

## 8. 出力ルール

- 上記ルールに反する実装は行わないこと。
- 既存実装がルールに違反している場合は、違反内容を明示した上でルール準拠の実装へ修正すること。
- ルール違反を回避できない場合は、実装を続行せず中断理由を明示すること。
- エラーメッセージ、エラーコード、表示文言を追加する場合は、必ず先に共通定数ファイルを作成または更新すること。
- 実装ファイル内に、エラーメッセージ、エラーコード、表示文言を `const` で直接定義してはならない。
- 既存コードにファイルローカルなエラーメッセージまたはエラーコード定義がある場合は、違反内容を明示した上で共通定義へ移設すること。
- 既存の共通定義が存在するにもかかわらず、同義の定数を重複追加してはならない。
- 変更内容を出力する際は、必要に応じて以下を明示すること。
  - 違反していた箇所
  - どの定数ファイルへ移設したか
  - 既存定義を再利用したか、新規追加したか
  - 中断した場合の理由

## 9. 具体禁止事項

- `UNKNOWN_ERROR` のようなエラーコードを hook / component / utility / service 内に直接書いてはならない。
- `エラーが発生しました。時間をおいて再度お試しください。` のような汎用エラーメッセージを、実装ファイル上部に直接書いてはならない。
- API エラー処理で使用するエラーコードおよびエラーメッセージは、必ず共通の `*.config.ts` に `enum` として定義し、実装側は import して利用すること。
- 文字列リテラルによるエラー種別判定を禁止する。
- ルール準拠のために必要な共通定義を作成せず、実装ファイルだけで完結させてはならない。

## 10. 優先適用ルール

- 「ファイル上部に定義してよい定数」よりも、「共通定義すべき値」の判定を優先すること。
- エラーメッセージ、エラーコード、表示文言は、ファイル上部に定義してよい定数の対象外とする。
- これらは必ず `*.config.ts` に `enum` で定義し、利用側では import して参照すること。

## 11. 後方互換対応の禁止

- re-export のみで構成されるファイル（`export { X } from "./X.config"` だけのファイル）を後方互換のために残してはならない。
- 未使用変数を `_` プレフィックスで温存してはならない。
- 削除済みコードに `// removed` コメントを残してはならない。
- 既存の import 先を変更したい場合は、シムを介さず全ての import 元を直接書き換えること。
