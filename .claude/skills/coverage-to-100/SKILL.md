# カバレッジを確定表の閾値まで満たす

> **呼び出し元**: `/implement-from-issue` の品質ゲート、または `/fix-implementation` のカバレッジ不足対応から呼ばれる補助 skill。**合格ラインの正典は各確定表**（バックエンド: `backend-00-stack.md` #13 = 命令(INSTRUCTION)100% / 分岐(BRANCH)90%、フロントエンド: `frontend-00-stack.md` #10 = 100%、いずれも除外後）。**分岐は 90% 基準で、到達不能分岐のための 100% 全域強制はしない**（緑化目的のノイズテストを避ける）。単独起動も可。

次の対象のカバレッジを改善する: $ARGUMENTS

## 指示

1. 最新のカバレッジレポートを見つける。未生成でプロジェクト設定済みなら生成する。
2. 未カバーの業務ロジック、分岐、異常系、入力検証を特定する。
3. 実質的なギャップを埋めるために必要最小限の意味あるテストを追加する。
4. 現状設計では妥当にテストできない場合のみリファクタを行う。
5. 再度カバレッジを計測し、残ギャップを報告する。**閾値判定は機械的に行う**: `bash .claude/skills/_common/scripts/check-coverage-threshold.sh <repo-type>`（backend | frontend）を実行し、exit 0 を確認する（このスクリプトは確定表 backend-00-stack.md #13 / frontend-00-stack.md #10 の閾値と JaCoCo/Istanbul を機械突合する。別担当が作成・本スキルは呼び出す前提）。exit 1 なら**ファイル/パッケージ単位で未達箇所を列挙**し、その箇所のテストを補う。
6. 生成コードや非本質的なフレームワーク接着層が閾値達成の障害なら、正当な最小除外として記録する（除外は pom の jacoco excludes に 1 行理由コメント付きで追加）。分岐は 90% 基準であり、残る到達不能分岐のためだけのテストは書かない。
