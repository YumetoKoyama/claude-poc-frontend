# 設計書から単体テストを作成する

> **パス解決（マルチリポジトリ対応）**: 本スキル内の `docs/requirements/`・`docs/design/`・`docs/test/` は **docs リポジトリ（claude-poc-docs）ルート相対**のパスを指す。
> - docs リポジトリをカレントとして実行している場合: そのまま使う。
> - 親アンブレラ（claude-poc-rules）から実行している場合（カレント直下に `claude-poc-docs/` が存在する場合）: これらすべてのパスに `claude-poc-docs/` を前置して読み書きする。
> - CI（子リポジトリ単体のチェックアウト）で docs リポジトリが存在しない場合: workflow が追加チェックアウトした docs のパスを使う。それも無い場合は Issue 本文に埋め込まれた設計情報を入力とし、原本の参照が必要なら中断して人間に確認する。

> **呼び出し元**:
> - **TDD ループの `test-write` 段から** `pre-write <Issue番号>` モードで呼ばれる（`/implement-loop` の新フロー）。この場合は実装コードが存在しないため、テストコードを生成するのみで実行しない。
> - `/implement-from-issue` の品質ゲート（手順 5・Pattern 2 並列ファンアウト）から「単体テスト」担当として呼ばれる（従来フロー）。
> - 設計フェーズ完了後に単独起動することもできる。
> - Agent Teams は使わない。

次の入力をもとに単体テストを作成または更新する: $ARGUMENTS

## モード判定

第 1 引数が `pre-write` の場合は **TDD 先行モード**、それ以外は **通常モード**として動作する。

| モード | 引数例 | 呼び出し元 | テスト実行 | 用途 |
|--------|--------|-----------|-----------|------|
| **TDD 先行モード** | `pre-write 42` | `implement-loop` の `test-write` 段 | **しない** | 実装前にテストコードを先出しする。テストは最初は失敗（または未コンパイル）が正常状態 |
| **通常モード** | `docs/design/api/jobs.yaml` | `implement-from-issue` 品質ゲート | する | 実装後にテストを実行し、失敗を修正する |

## 指示

### 共通手順（モード共通）

1. 設計書または要件定義書を読む。第 1 引数が `pre-write` の場合は第 2 引数を Issue 番号として `gh issue view <Issue番号>` で Issue を取得し、関連設計書を特定する。
   - 要件定義の受け入れ条件: `docs/requirements/functional/[機能名].md` の AC-XXX
   - API の仕様: `docs/design/api/[リソース名].yaml`（リソースごとに 1 ファイル、共通スキーマは `docs/design/api/_common.yaml`）
   - 画面の仕様: `docs/design/screens/[scr-id]-*.md`
   - DB の仕様: `docs/design/tables/[テーブル名].md`
   - テストケースマトリクス（`test-design` 段で生成済みの場合）: `docs/test/単体テストマトリクス.md`

2. 既存テストファイルを Glob/Grep で確認する。既にテストコードが存在する場合は内容を確認した上で差分のみ追加・更新する（上書き禁止）。

3. 要件の AC-XXX、業務ルール BR-XXX、API の各レスポンス、入力検証、分岐、例外処理に対して焦点のあるテストを追加する。バックエンドは JUnit 5 + Mockito + MockMvc、フロントエンドは Vitest + React Testing Library を使う。各テストメソッドのコメントまたは `@DisplayName` に対応する **TC-XXX** を明記する。
   - **必ず網羅する区分**: 正常系 / 異常系（入力エラー）/ 境界値 / **権限境界**。
   - **状態遷移・並行制御（RC-01）**: 状態遷移エンドポイントは、楽観/悲観ロック競合時に 409 となること・不正遷移が拒否されることをテストする。
   - **認可・テナント分離（RC-03）**: 各 operationId の `@PreAuthorize` がロール限定どおりに効くこと、テナント越境が 404 になること、テナントフィルタが **UPDATE/DELETE/COUNT/集計**でも効くこと。
   - **契約・データ連鎖（2026-06-12）**: BE Response DTO の JSON フィールド名（Jackson 反映後）が OpenAPI スキーマと一致すること。FE は API レスポンスのフィールドを実際に使用し、空文字・固定値で握り潰さない（dead-field 回帰テスト）。

### TDD 先行モード（`pre-write`）のみの追加手順

4. テストコードを生成した後、**テストを実行しない**。実装コードが存在しないため、テストはコンパイルエラーまたは実行失敗になることが想定どおりの状態である。
5. テストファイルの冒頭コメントに `// TDD: pre-write — 実装前テスト。/implement-from-issue 実行後に全テストが通ることを確認する` と明記する。
6. テストコードを feature ブランチにコミットする（実装コードはまだ存在しないため別途コミット不要）:
   ```bash
   git add <テストファイルパス>
   git commit -m "test(#<Issue番号>): TDD 先行テストコード生成（実装前）"
   git push || true
   ```

### 通常モード（品質ゲート呼び出し）のみの追加手順

4. 最も狭い関連テストコマンドを実行する。
5. テスト失敗時は、アプリケーションコード・テスト設計・セットアップのどこに原因があるかを判断し、根本原因を修正する。実装コードに起因する場合はアプリケーションコードを修正する（テストコードを弱めることで緑化することは禁止）。
6. 修正ごとに同じ焦点の検証を再実行する。
